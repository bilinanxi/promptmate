# Windows AI 提示词助手完整开发方案

> **For Hermes:** 实施时按里程碑逐项开发；核心模块采用 TDD，完成每个里程碑后进行规格检查、自动化测试和人工验收。

**目标：** 开发一款面向新手与轻度创作者的 Windows 本地开源提示词助手，覆盖文生图、图生图、文生视频、图生视频、多模型优化、双语输出、图片反推、语义匹配、词库管理、自定义 API 与悬浮窗自动粘贴。

**产品形态：** Windows 10/11 本地桌面应用；主窗口与悬浮窗并存；无需账号、业务服务器或远程更新服务。

**建议技术栈：** Tauri 2、React、TypeScript、Vite、Rust、SQLite、JSONL、YAML、Vitest、Playwright、Cargo Test。

---

## 1. 已确认的产品决策

### 1.1 用户定位

- 主要服务刚接触 AI 图片/视频生成的新手。
- 同时服务有一定基础、但不愿手工编写繁琐提示词的用户。
- 不以提示词专家或复杂节点工作流用户为主要对象。
- 核心价值是降低理解成本、减少平台切换和提高提示词复用效率。

### 1.2 第一版范围

第一版即实现完整设想：

- 文生图、图生图、文生视频、图生视频；
- 原文本输入、词库拼装和纯词库拼装；
- 不同目标模型的提示词优化；
- 中文、英文和中英混合输入；
- 始终生成中文、英文两份结果；
- 模糊输入联想、补全和优化；
- 忠实、均衡、创意三个补全档位；
- 图片反推、结构化拆解和语义匹配；
- 缺失词条自动生成、分类并立即可用；
- 内置、自定义、导入、AI 生成内容的来源区分；
- Bing 搜索提示词解释和参考资料；
- 严格格式的 JSONL/CSV/XLSX 导入导出；
- 服务商预设、自定义 OpenAI-compatible API、Ollama；
- 多 API 配置及主备自动切换；
- Windows 主窗口、悬浮球、智能优化悬浮面板和词库快选悬浮面板；
- 自动粘贴到用户当前网站输入框。

### 1.3 不包含内容

- macOS、Linux、移动端；
- 用户账号、云同步、在线社区；
- 远程配置中心和自动更新服务；
- 图片或视频生成任务本身；
- 托管公共 API Key；
- 浏览器扩展。

## 2. 核心用户流程

### 2.1 智能优化模式

1. 选择图片/视频任务类型。
2. 选择平台和具体模型。
3. 输入中文、英文或混合原始描述；原始描述可为空。
4. 从右侧词库补充人物、场景、动作、镜头、风格等词条。
5. 选择忠实、均衡或创意档位。
6. 调用文本模型，根据模型档案生成结构化卡片和中英文提示词。
7. 用户删除、替换、锁定或调整卡片顺序。
8. 用户切换查看中文/英文结果，手动修改、复制或保存为模板。

### 2.2 快速拼装模式

1. 选择目标平台和模型。
2. 在分类词库中搜索或点选词条。
3. 软件按模型档案规定的字段顺序和分隔符即时组合。
4. 全程不调用 AI，可离线使用。
5. 用户切换中文/英文结果并复制。

### 2.3 图片反推模式

1. 拖入、粘贴或选择本地图片。
2. 明确显示当前使用本地视觉模型还是远程视觉接口。
3. 视觉模型输出严格结构化结果。
4. 将结果拆分为主体、服装、动作、场景、灯光、构图、风格等概念。
5. 使用规范化、同义词、双语文本和 Embedding 进行语义匹配。
6. 匹配成功时复用已有概念；匹配失败时自动生成新概念。
7. 新概念自动分类、标记为 `ai_generated` 并立即可用。
8. 一个概念只保存一次，通过模型变体关联多个模型表达。
9. 输出适配当前模型的完整中英文提示词，并可发送到主编辑器。

### 2.4 悬浮窗模式

- 悬浮球可拖动、置顶、吸附屏幕边缘和贴边收起。
- 点击后显示两个页签：`智能优化`、`词库快选`。
- 智能优化页包含输入、平台/模型、补全档位、双语切换、优化和复制。
- 词库快选页包含分类、搜索、最近使用、收藏和双语切换。
- 用户先在目标网站提示词输入框定位光标，再打开悬浮窗。
- 点击词条后恢复目标窗口焦点，在光标处追加提示词。
- 分隔符由模型档案决定；支持撤销上一次插入。
- 自动粘贴失败时保留到剪贴板并提示手动粘贴。
- 设置中允许关闭自动粘贴，退回只复制模式。

## 3. 信息架构与界面

### 3.1 主窗口

采用 Windows 11 Fluent 风格，支持浅色、深色和跟随系统主题。

主工作台为可收缩三栏：

- 左栏：功能导航、任务类型、平台/模型、最近项目。
- 中栏：模式切换、原始输入、已选卡片、优化按钮、双语结果。
- 右栏：分类树、关键词/语义搜索、词库、联想和推荐。

左右栏可独立收起；窄窗口下右栏变为抽屉。

### 3.2 独立页面

- 图片反推；
- 词库管理；
- 分类和标签管理；
- 模型档案管理；
- API 与本地模型设置；
- 导入导出；
- 模板、收藏和历史；
- 外观、隐私与悬浮窗设置。

### 3.3 来源视觉标识

来源不能只依赖颜色，还要显示图标或文字，保证色觉障碍用户可辨别：

- `builtin`：内置，只读；
- `user`：用户自定义；
- `imported`：外部导入；
- `ai_generated`：AI 自动生成。

## 4. 分类体系

### 4.1 顶层类型

- 图片提示词；
- 视频提示词；
- 图片/视频通用概念。

底层使用统一概念记录，通过 `media_types` 标记适用范围；界面上按图片和视频分别浏览，避免物理重复。

### 4.2 图片标准分类

- 主体与人物种类；
- 性别、年龄与身份；
- 外貌、发型与妆容；
- 服装与配饰；
- 动作与姿态；
- 表情与视线；
- 场景与环境；
- 道具与背景元素；
- 艺术与视觉风格；
- 灯光；
- 机位与视角；
- 景别与镜头；
- 构图；
- 色彩与调色；
- 滤镜；
- 特效；
- 材质与细节；
- 画质；
- 分辨率与画幅；
- 负面提示词。

### 4.3 视频扩展分类

- 分镜与镜头段落；
- 摄影机运镜；
- 主体运动；
- 环境运动；
- 动作过程；
- 时间变化；
- 速度与节奏；
- 镜头衔接与转场；
- 首帧与尾帧；
- 时长与帧率；
- 主体/角色一致性；
- 音效、环境声、对白和音乐；
- 视频负面提示词。

### 4.4 分类管理规则

- 内置标准分类可隐藏但不可删除。
- 用户可新增分类、子分类和标签。
- 用户分类可修改、移动、隐藏和删除。
- 无法确定分类的自动生成内容进入“待分类”。
- 删除用户分类前必须选择迁移内容或一并删除。

## 5. 数据设计

### 5.1 数据源与缓存

- JSONL/YAML 是可移植、可编辑的源数据。
- SQLite 是搜索、关系、历史和向量索引缓存。
- 应用启动或文件变更后增量重建索引。
- 内置数据安装在应用资源中，只读。
- 用户数据写入 `%APPDATA%/<AppName>/data`，避免 Windows 安装目录权限问题。

### 5.4 导入导出

- JSONL：完整无损交换格式。
- CSV：适合 Excel/WPS 批量编辑。
- XLSX：第一版直接支持，提供字段说明工作表和数据验证示例。
- 所有格式包含 `schema_version`。
- 导入分为解析、Schema 校验、业务校验、重复检查、预览、事务写入六步。
- 错误必须报告文件、工作表、行号、字段、错误值和修复建议。
- 导入失败整体回滚。
- 冲突策略：跳过、覆盖、创建副本、逐项处理。

### 5.5 GitHub 开源提示词采集管线

GitHub 项目作为内置词库、模型规则和分类词表的重要候选来源，但不得直接抓取后无审查地打包发布。

采集流程：

1. 通过 GitHub Search、Topics 和人工推荐建立候选仓库清单。
2. 固定到具体 Commit，记录仓库、文件路径、作者、许可证和访问日期。
3. 检查仓库许可证、数据子目录附加许可证、README 署名要求和第三方内容声明。
4. 许可证为 Apache-2.0、MIT、BSD、CC0 等兼容类型时进入内容审核；GPL/AGPL/CC-BY/CC-BY-SA 等由发布许可证兼容性审查后决定；没有明确许可证时默认不分发。
5. 只解析提示词、词表、模型指南等目标数据，忽略网页代码、密钥、可执行脚本和仓库内嵌指令。
6. 对内容执行去重、语言识别、敏感内容标记、模型识别和分类拆解。
7. 翻译或改写时保留原文，不把机器翻译伪装成原作者原文。
8. 人工抽样复核后转换为项目 JSONL Schema。
9. 生成 `THIRD_PARTY_DATA.md`、NOTICE 和机器可读来源清单。
10. 每条数据保留 `provenance`；用户可在词条详情中打开固定 Commit 的来源页面。

候选源状态分级：

- `approved`：许可证兼容、内容已审核，可以随软件分发；
- `review`：许可证或内容仍需人工复核；
- `reference_only`：仅用于理解分类和模型规则，不复制内容；
- `rejected`：授权不明、来源混杂、重复灌水或质量不足。

首轮调研已发现的候选项目示例（发布前需再次核验具体文件授权）：

- `PicoTrex/Awesome-Nano-Banana-images`：GitHub API 显示 Apache-2.0；图片提示案例候选。
- `ZizheRuan/awesome-nanobanana2`：GitHub API 显示 Apache-2.0；中英文图片提示案例候选。
- `cuigh/awesome-nano-banana-prompts`：GitHub API 显示 Apache-2.0；图片提示案例候选。
- `SoraEase/sora-prompt`：GitHub API 显示 CC0-1.0，仓库已归档；Sora 视频提示案例候选。
- `Creatify-AI/ai-ad-prompt-guide`：GitHub API 显示 MIT；图片/视频模型规则候选。
- `smixs/visual-skills`：GitHub API 显示 MIT；多模型提示规则候选。
- `geekjourneyx/awesome-ai-video-prompts`：搜索结果声明 MIT，仍需读取仓库许可证确认；视频指南候选。

## 6. 多模型适配

### 6.1 档案机制

- 平台与模型采用两级选择。
- 内置档案只读。
- 用户可复制内置档案后编辑。
- 用户可创建、导入、导出自定义档案。
- 提供通用图片模型和通用视频模型兜底。
- 模型列表支持常用、全部、最近使用和收藏。

### 6.2 第一版覆盖原则

实施时以官方最新资料核验具体版本，不在核心代码硬编码版本名。初始平台族至少覆盖：

- 国际图片：OpenAI 图像、Midjourney、Google 图像、FLUX、Stable Diffusion、Ideogram、Recraft、Adobe Firefly；
- 国内图片：即梦/Seedream、通义万相/Wan、腾讯混元图像、可灵图片及常见平台；
- 国际视频：Sora、Veo、Runway、Luma、Pika；
- 国内视频：可灵、即梦/Seedance、海螺、Vidu、通义万相/Wan、PixVerse 等。

产品发布前必须再次对照官方文档确认名称、支持任务和提示词语法。

## 7. AI 与语义能力

### 7.1 能力拆分

- 文本：补全、优化、翻译、结构化、分类和模型适配。
- 视觉：图片理解和反推。
- Embedding：语义检索、近义匹配和去重。

三类能力分别配置，不假设同一服务商同时支持全部能力。

### 7.2 补全档位

- 忠实：纠错、去重、消歧，不扩展创意。
- 均衡：保留核心，适量补充镜头、灯光、构图和质量。
- 创意：允许扩展服装、动作、环境、天气和氛围。

AI 新增卡片显示 `AI 补充` 标识，并允许删除或锁定。

### 7.3 双语规则

- 自动识别中文、英文和混合输入。
- 每次优化始终生成中英文两份。
- 两份结果语义一致，但允许按目标模型习惯重排，禁止机械逐字翻译。
- 中文和英文用页签切换并分别复制。
- 手动修改一份时提示是否同步重建另一份。

### 7.4 语义匹配管线

1. Unicode 和大小写规范化。
2. 精确匹配 ID、标准名称和别名。
3. 分词与关键词匹配。
4. 中英文 Embedding 召回候选。
5. 类别、媒体类型和模型适用性加权。
6. 高置信度自动复用。
7. 临界置信度由文本模型二次判断。
8. 低置信度自动创建 `ai_generated` 概念。

阈值必须配置化，并使用固定测试集调优，不能凭感觉写死。

## 8. API 与隐私

### 8.1 接口体系

- 服务商预设；
- 自定义 OpenAI-compatible Base URL、API Key、模型名和参数；
- Ollama 本地服务自动检测；
- 支持多个配置档案；
- 文本、视觉、Embedding、翻译分别配置主接口和备用接口；
- 超时、限流、额度不足或模型不可用时，提示后自动切换。

### 8.2 安全规则

- API Key 使用 Windows Credential Manager，不写入普通配置、日志或导出文件。
- 日志默认不保存完整提示词和图片内容。
- 图片发送远程服务前显示服务商和隐私提示。
- 视觉主接口失败后，第一次切换到另一个远程服务必须征得同意。
- 本地模型清楚显示“数据不离开本机”。
- 提供清除历史、缓存、临时图片和凭据功能。

### 8.3 离线降级

未配置 API 时仍可使用：

- 快速拼装；
- 精确与关键词搜索；
- 双语词库；
- 分类和标签；
- 导入导出；
- 模板、收藏和历史；
- Bing 跳转。

语义匹配可使用本地 Embedding 模型；本地模型未安装时降级为词法和别名匹配，并明确提示能力差异。

## 9. Windows 悬浮窗与自动粘贴

### 9.1 技术策略

- 使用 Tauri 独立 `WebviewWindow` 实现悬浮球和弹出面板。
- 使用 Rust `windows` API 记录前台窗口和焦点控件。
- 插入前再次验证目标窗口，防止写入错误窗口。
- 对中文及复杂 Unicode 文本采用临时剪贴板加粘贴方式。
- 保存并恢复原剪贴板内容；恢复失败时提示。
- 使用模型档案定义追加分隔符。
- 每次插入记录可撤销操作，仅在本地短期保存。

### 9.2 边界与降级

- 某些浏览器页面、管理员权限窗口或受保护控件可能拒绝自动粘贴。
- 普通权限应用不得向更高权限窗口注入输入。
- 自动粘贴失败时复制文本并显示可操作提示。
- 提供全局快捷键，但允许用户修改或禁用。
- 自动粘贴、剪贴板恢复、开机启动和置顶均为独立开关。

## 13. 风险与处理

### 13.1 首版范围大

处理：按 M0–M10 逐个形成可运行版本，但不提前公开缩水版；每个里程碑都做回归测试。

### 13.2 常见平台规则变化快

处理：数据驱动模型档案；发布前核验官方资料；用户可本地导入更新，不依赖远程更新系统。

### 13.3 AI 自动内容污染词库

处理：统一概念 ID、混合语义去重、来源标识、删除与历史恢复；自动内容不伪装成内置内容。

### 13.4 自动粘贴兼容性

处理：聚焦普通桌面/浏览器文本框；权限和焦点双重检查；失败时退回复制，不承诺绕过系统安全边界。

### 13.5 免费 API 不稳定

处理：不承诺第三方永久免费；支持免费额度、多个服务商、自定义接口、本地 Ollama和主备切换。

### 13.6 词库建设工作量

处理：先建立高频、高质量核心词库；使用格式校验和贡献指南接受社区扩展；内置内容发布前人工复核。

## 14. 难度与周期判断

- 整体难度：中等偏高，图片反推、跨语言语义去重和 Windows 自动粘贴属于高难模块。
- 单人全职开发并达到可公开发布质量，合理预期为数月级项目，而不是几天或一两周的小工具。
- 最大非代码工作量是常见模型档案、双语词库和兼容性测试。
- HTML 原型只能验证信息架构、视觉和主要交互，不能证明 Windows 自动粘贴、凭据存储或真实 API 兼容性。
