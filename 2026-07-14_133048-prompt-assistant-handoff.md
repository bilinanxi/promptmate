## 一、项目概述

计划开发一款**开源、本地运行的桌面提示词助手**，面向文生图、文生视频、图生图和图生视频场景。

产品不直接负责生成图片或视频，第一阶段主要负责：

- 管理、分类和组合提示词；
- 支持中英文输入与输出；
- 根据目标生成模型优化提示词；
- 对模糊描述进行联想、补全和专业化改写；
- 允许用户配置免费额度 API、OpenAI Compatible API 或本地模型；
- 保存、导入和导出独立的提示词库文件；
- 后续通过视觉模型分析图片并生成、匹配和归类提示词。

建议产品形态：**Windows 优先的本地桌面应用**。候选技术方案为 Tauri + Vue/React；若优先验证原型，可考虑 Python + PySide6。

---

## 二、已确认的原始需求

1. 针对不同图片、视频模型进行提示词优化。
2. 支持中文、英文以及中英混合输入。
3. 对模糊输入自动联想、优化和补全。
4. API 由用户自定义配置，软件不依赖自建中心服务器。
5. 将提示词按图片、视频及人物、分镜、运镜、风格、画质、服装、场景、动作、表情、灯光、机位、构图、道具、滤镜、特效等维度分类。
6. 软件开源、本地运行，不要求远程更新。
7. 用户自定义提示词必须与软件内置提示词明确区分。
8. 图片分析生成提示词后，与本地词库进行语义匹配；找不到合适词条时生成新候选并归类。
9. AI 生成词条必须与内置和用户词条有不同的颜色、图标或文字标记。
10. 提示词解释通过系统浏览器打开 Bing 搜索结果。
11. 提示词以独立文件保存，规定正式格式，并支持导入、导出。

## 三、已形成的方案共识

### 1. 数据来源必须分开

每条词条使用 `source` 标记来源：

- `builtin`：软件内置，只读；
- `user`：用户创建，可编辑；
- `ai_generated`：AI 生成，默认待确认。

界面不能只靠颜色区分，还应显示图标或文字，避免无障碍问题。

### 2. 提示词与生成参数分开

应区分：

- 提示词文本；
- 模型生成参数，例如宽高比和分辨率；
- 软件内部元数据，例如分类、标签、来源和版本。

### 3. API 采用适配层

第一版至少支持：

- OpenAI Compatible API；
- 自定义 Base URL；
- 自定义 API Key；
- 自定义文本模型和视觉模型名称；
- 接口连通性测试；
- 本地 Ollama/LM Studio 等兼容服务。

软件可以免费开源，但不能承诺第三方 API 永久免费。

### 4. 图片“反推”应准确命名

建议使用“图片分析并生成提示词”或“图片转提示词”，不要承诺恢复原作者的真实提示词。

AI 新词条不应静默写入正式词库，应先进入“待确认”状态，由用户确认、修改、合并或拒绝。

### 5. 主数据格式采用 JSON

建议将内置、用户、AI 生成的数据分别保存，避免安装或升级时互相覆盖。

示例目录：

```text
prompt-library/
├── manifest.json
├── image/
│   ├── character.json
│   ├── clothing.json
│   ├── lighting.json
│   └── composition.json
├── video/
│   ├── storyboard.json
│   ├── camera-movement.json
│   └── transitions.json
├── user/
└── generated/
```

单条词条建议结构：

```json
{
  "id": "lighting_soft_cinematic_001",
  "type": "fragment",
  "mediaType": ["image", "video"],
  "category": "lighting",
  "name": {
    "zh-CN": "柔和电影灯光",
    "en": "soft cinematic lighting"
  },
  "prompt": {
    "zh-CN": "柔和的电影感灯光",
    "en": "soft cinematic lighting"
  },
  "description": {
    "zh-CN": "降低阴影反差，营造柔和的电影氛围",
    "en": "Creates a soft cinematic atmosphere with reduced shadow contrast"
  },
  "tags": ["柔光", "电影感", "人像"],
  "supportedModels": ["generic", "flux", "sdxl"],
  "source": "builtin",
  "status": "approved",
  "version": 1
}
```

## 五、MVP 建议范围

首版只做以下闭环：

1. 图片、视频两个本地分类提示词库；
2. 打开软件即展示丰富词条和新手灵感组合；
3. 分类浏览、搜索、标签筛选、收藏和最近使用；
4. 点击卡片加入灵感篮、排序、移除、清空和自动拼装；
5. 中英文词条及中英文结果切换；
6. 内置、自定义、导入、AI 生成来源区分；
7. JSONL/CSV 导入导出；
8. 可选的 OpenAI-compatible/Ollama 通用翻译、分类和润色；
9. 系统浏览器打开 Bing 搜索。

首版暂缓：

- 图片分析与语义匹配；
- 分镜时间轴；
- 批量图片处理；
- 云同步、登录和用户系统；
- 直接生成图片或视频；
- 平台选择、目标生成模型选择和专用模型适配规则。

## 六、MVP 验收标准

- [ ] 启动后无需输入文字即可看到图片或视频词库、分类和推荐组合。
- [ ] 无网络时仍可浏览、搜索、筛选、收藏、组合、编辑和导入导出本地提示词。
- [ ] 点击词条卡片本体即可加入或移出灵感篮，鼠标和键盘操作结果一致。
- [ ] 用户只需选择图片或视频，不需要选择平台或目标生成模型。
- [ ] 用户可以配置任意 OpenAI-compatible Base URL、Key 和 AI 服务模型名称，且该模型名不影响词库分类。
- [ ] API 不可用时给出明确错误，不丢失用户原始输入。
- [ ] 不配置 API 时仍能从词库自动拼装并复制中文或英文提示词。
- [ ] 可选 AI 润色不会遮挡或替代词库浏览和离线拼装主流程。
- [ ] 内置词条不可被直接覆盖，用户可复制后自定义。
- [ ] 内置、自定义、导入和 AI 生成来源在颜色、图标和文字上均可区分。
- [ ] 导出的完整词库可在全新安装环境中重新导入。
- [ ] 导入过程可处理 ID 冲突、重复内容、缺失字段和不支持的格式版本。
- [ ] API Key 不出现在日志、导出包或错误详情中。
- [ ] 点击解释按钮会使用默认浏览器打开正确编码的 Bing 搜索地址。
- [ ] Windows 安装包能在干净环境中完成安装、运行和卸载。

## 七、主要风险与处理建议

| 风险 | 影响 | 建议 |
|---|---|---|
| 项目范围过大 | 首版长期无法完成 | 首版聚焦图片/视频词库与离线拼装，图片分析放到第二阶段 |
| 免费 API 不稳定 | 核心 AI 功能不可用 | 用户自配接口，同时支持本地兼容服务 |
| 词库分类过多 | 新手仍然难以选择 | 提供推荐组合、标签筛选、搜索和渐进式分类浏览 |
| 内置词库质量差 | 搜索和组合体验差 | 少而精、人工审核、去重、注明许可 |
| AI 自动生成污染词库 | 重复和错误持续累积 | 新词条进入待确认区，禁止静默正式入库 |
| 图片上传隐私 | 用户敏感内容泄露 | 调用第三方 API 前提示；支持本地视觉模型 |
| 只靠颜色区分来源 | 无障碍体验差 | 同时使用图标和文字标签 |
| “图片反推”承诺过度 | 用户预期错误 | 命名为“图片转提示词”，说明仅分析可见语义 |
